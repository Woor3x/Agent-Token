# 项目分工方案

## 0. 阅读指南

- 按**模块 (M)** 分工，每模块含若干**任务 (T)**
- 任务字段: `[id] 标题 [优先级][规模][依赖]`
  - 优先级: **P0** (红线/必做) / **P1** (核心加分) / **P2** (优化)
  - 规模: **S** (≤1d) / **M** (2-3d) / **L** (≥4d)
  - 依赖: 其他任务 id 或 `-` 无
- 所有人共同: 代码 review + 文档编写 + 演示准备
- 认领后在 "认领表" 填名字

## 1. 模块清单

| 模块               | 代号     | 核心职责                                                    | 主要技术                              |
| ---------------- | ------ | ------------------------------------------------------- | --------------------------------- |
| IdP 身份与授权颁发      | **M1** | Client assertion / Token Exchange / Revoke / JWKS / KMS | Python, PyJWT, cryptography, age  |
| Gateway (PEP)    | **M2** | AuthN / Intent / AuthZ / One-shot / Trace / Audit       | FastAPI, httpx, OPA client, Redis |
| OPA 策略 + 异常检测    | **M3** | Rego 策略 / 数据 / anomaly detector                         | Rego (OPA), asyncio               |
| Agents (3 个)     | **M4** | DocAssistant (orchestrator) / DataAgent / WebAgent      | LangGraph, 飞书 SDK, LLM            |
| SDK + 测试套件       | **M5** | agent\_token\_sdk / 单元 + 集成 + 演示测试                      | Python SDK, pytest                |
| Web 前端 + 可视化     | **M6** | 聊天 UI / DAG / Trace / 审计 / 撤销面板                         | Next.js, React Flow, Tailwind     |
| DevOps + 文档 + 演示 | **M7** | docker-compose / start.sh / README / 演示脚本 / 视频          | Docker, shell                     |

## 2. 推荐团队规模

建议 4-6 人。典型分工:

| 人数  | 推荐合并                               |
| --- | ---------------------------------- |
| 3 人 | A=M1+M3, B=M2+M5, C=M4+M6+M7       |
| 4 人 | A=M1, B=M2+M5, C=M3+M4, D=M6+M7    |
| 5 人 | A=M1, B=M2, C=M3, D=M4+M5, E=M6+M7 |
| 6 人 | 一模块一人                              |

## 6. 契约协同规则

- **接口先行**: M1 和 M2 之间、M2 和 M4 之间 **先签 OpenAPI/JSON Schema**，再并行开发
- **Mock 优先**: 依赖方可先用 mock，解依赖
- **每模块一个 OWNER**: 负责 API 稳定性 + code review 最终拍板
- **共享类型**: `schemas/` 目录放 pydantic 模型 + JSON schema，所有模块 import
- **审计事件 schema**: 由 M3 OWNER 拟定，M1/M2 follow
- **错误码**: 统一在 `schemas/errors.py` 定义，跨模块共用
